# مواصفات التبويب 5: الشراكات

**النسخة:** 1.0 للاعتماد · **تبويب جديد بالكامل** (يرث بيانات الشراكات الحالية بالترحيل المعتمد).
**مبدأ حاكم:** الشراكة علاقة تسبق المشاريع وتبقى بعدها — تجيب: مع من نتعامل؟ بأي اتفاق؟ كم دُفع؟ · مسؤولها دور «مدير الشراكات» (السابع — مُسند حاليًا لمدير المشاريع).

---

## 1. الجهات الشريكة
**السجل:** الاسم الرسمي، النوع (جمعية تحفيظ/مدرسة/شركة تعليمية/وقف/جهة حكومية/أخرى)، المدينة، **الدور** (متعاقدة/جهة تنفيذ/مانحة — قد تجمع أكثر من دور)، مسؤولو التواصل (متعدد: الاسم، الصفة، الجوال، البريد — أحدهم رئيسي)، ملاحظات.
**صفحة الجهة الجامعة:** البيانات · شراكاتها (كل الرحلات) · مشاريعها عبر الشراكات · **سجل الأثر المتراكم** (يتجمع من قياسات المشاريع تلقائيًا) · مستنداتها (عقود، عروض، تقارير ختامية) · خط زمني للتواصل (لقاءات ومحاضرها).
**قاعدة:** الجهة سجل دائم لا يُحذف — والشراكات تأتي وتذهب تحته.

## 2. رحلة الشراكة السباعية
**المراحل والحالات:**
```
1. فرصة ← 2. تواصل ← 3. لقاء/عرض تعريفي ← 4. تشخيص الاحتياج ← 5. عرض السعر ← 6. تعاقد ← 7. تنفيذ
                                                                                    ↘ متعثرة (من أي مرحلة، بسبب)
النهايات: قياس وتجديد (رحلة جديدة مرتبطة) أو إغلاق (بسبب: اكتمال/عدم تجديد/إلغاء)
```
**العرض:** لوحة أعمدة (Kanban) بمراحلها + عرض قائمة — البطاقة: الجهة، البرامج محل الاهتمام، القيمة المتوقعة، مسؤول المتابعة، عمر المرحلة (تمييز الراكدة > N يومًا).
**لكل مرحلة نموذجها:**
- **فرصة:** المصدر (ملتقى/توصية/تواصل مباشر/تسويق)، البرنامج محل الاهتمام، الخطوة القادمة وتاريخها.
- **لقاء:** موعد + محضر مرفق + الاحتياج الأولي.
- **تشخيص:** إرسال استبانة عبر الرابط (الفئة، الأعداد، البيئة) + نتيجة القياس القبلي إن أُجري + الخدمات المقترحة.
- **الانتقال بين المراحل:** يدوي بيد مدير الشراكات مع ملاحظة، ويُسجَّل (من/متى) — والتعاقد وحده انتقال آلي (باكتمال شروطه أدناه).

## 3. عرض السعر
**البناء:** من الرحلة — اختيار البرامج من مكتبة المشاريع (الأسعار تُسحب من بطاقة البرنامج) + بنود الخدمات (حقائب بالكمية، تدريب، زيارات بالعدد, استشارات، استبانات، تقارير) + خصم اختياري + **الضريبة تُحسب بنسبة الإعدادات** ← الإجمالي آلي (لا إدخال يدوي).
**الدورة:** مسودة ← معتمد داخليًا (صلاحية `partnerships.quotes.approve`) ← مرسل للجهة (عبر الرابط) ← بملاحظات الجهة (تعديل = **نسخة جديدة** والقديمة محفوظة) ← مقبول ← تحوّل لعقد.
**الإخراج:** PDF بهوية حلل وبيانات الشركة (المُدارة هنا في الشراكات — قرار عبدالله).

## 4. العقد
يتولد من العرض المقبول (بنوده = بنود العرض) + مدة الشراكة + التزامات الطرفين + جدول الدفعات (دفعة/دفعتان/جدول) ← PDF نهائي ← **التوقيع بالرفع اليدوي فقط (قرار عبدالله):** الجهة تنزّل العقد من رابطها ← توقّعه وتختمه ← ترفعه ← مدير الشراكات يطابق ويؤكد ← **شرطا «تعاقد»: نسخة موقعة مؤكدة + تأكيد المالية للدفعة الأولى** (إن كان العقد يشترطها) ← تنشط أزرار «توليد مشروع».

## 5. الرابط الفريد للجهة
**الطبيعة:** رابط برمز طويل + **صلاحية زمنية من الإعدادات** + إبطال/تجديد يدوي — ترقية للرابط السحري القائم، مربوط بالشراكة (لا بجهة عامة).
**ما تراه الجهة عبر مراحل رحلتها:** استعراض البرامج المتاحة (بطاقات من المكتبة — دون أسعار داخلية) واختيار اهتمامها ← تعبئة استبانة التشخيص ← عرض السعر (قبول/ملاحظات) ← العقد (تنزيل/رفع الموقع) ← **تسجيل الدفعات** (المبلغ، إثبات التحويل — «بانتظار تأكيد المالية») ← بعد التوليد: مهامها من خطة الجهة، رفع شواهدها، مواعيدها، التقرير الختامي عند التسليم.
**قواعد أمنية صارمة:** الرابط يرى شراكته فقط — عزل مطلق عن بيانات حلل والجهات الأخرى · rate-limit · كل فعل عبر الرابط يُسجَّل في سجل النشاط منسوبًا للجهة · لا صلاحيات كتابة إلا المذكورة نصًا.
**كل حالة تظهر فورًا لمدير الشراكات** (إشعار: الجهة اختارت/قبلت/رفعت/دفعت).

## 6. سجل الدفعات
جدول دفعات العقد (المستحقة بتاريخها) مقابل المسجّلة: الجهة تسجّل عبر الرابط أو المدير يدويًا ← «بانتظار تأكيد المالية» ← تأكيد المالية ← **تُنشأ تلقائيًا كإيراد مؤكد في المالية** (مرة واحدة، مرجع مزدوج) ← الفاتورة الضريبية من نظام الفوترة تُرفع هنا مستندًا (قرار عبدالله). تنبيه الدفعات المتأخرة لمدير الشراكات.

## 7. زر «توليد مشروع» (نقطة العبور)
بعد التعاقد: زر لكل برنامج في العقد ← يفتح ملخص التوليد (البرنامج، قالباه، الخدمات المشمولة من العرض [تحدد البنود المتولدة]، تاريخ الانطلاق، مدير المشروع المقترح) ← تأكيد ← يُنشأ المشروع في تبويب المشاريع مربوطًا (`partnership_id`) — والتفاصيل في مواصفة المشاريع.

## قاعدة البيانات
`organizations` + `organization_contacts` · إعادة بناء `partnerships`: `organization_id`، المرحلة، مسؤول المتابعة، سبب التعثر/الإغلاق، مرجع التجديد · `partnership_stage_logs` · `quotes` + `quote_items` (بإصدارات) · `partnership_contracts` + `contract_payments_schedule` · `partnership_payments` (المسجلة، الحالة، مرجع الإيراد) · `partner_links` (الرمز، الصلاحية، الحالة) · `company_profile` (بيانات الشركة للعروض والعقود) · **الترحيل المعتمد:** استخراج الجهات من entity_name، عكس العلاقة، إسقاط الحالات الأربع على الرحلة، بأمر --dry-run واختبارات.

## الصلاحيات الهرمية
`partnerships.*` — الأقسام: `partnerships.organizations.[view|manage]` · `partnerships.pipeline.[view|manage]` · `partnerships.quotes.[view|create|approve]` · `partnerships.contracts.[view|manage|confirm]` · `partnerships.payments.[view|record]` · `partnerships.links.manage` · `partnerships.company-profile.manage` — الدور السابع «مدير الشراكات» يجمعها كلها.

## التكاملات
المشاريع (التوليد + البطاقات المتبادلة + صعود الأثر لسجل الجهة) · المالية (الدفعة المؤكدة ← إيراد) · الاجتماعات (اجتماع بنوع «شراكة» يظهر في صفحتها) · المستندات (عقود وعروض الجهة بتصنيفها) · الرئيسية (شراكات راكدة، دفعات متأخرة، نسخ موقعة بانتظار التأكيد في «يحتاج تدخلك»).

## الإشعارات
فعل الجهة عبر الرابط (اختيار/استبانة/قبول عرض/رفع عقد/تسجيل دفعة) · دفعة بانتظار تأكيد المالية · دفعة متأخرة · شراكة راكدة بمرحلتها · عقد شراكة يقارب الانتهاء (تجديد؟).

## الاختبارات المطلوبة (عناوين)
عزل الرابط الفريد (IDOR: رابط جهة لا يصل لشراكة أخرى أبدًا) · انتهاء وإبطال الرابط · لا «تعاقد» دون نسخة موقعة مؤكدة (+ دفعة إن اشتُرطت) · الإجمالي والضريبة آليان ونسخ العروض محفوظة · الدفعة المؤكدة تنشئ إيرادًا واحدًا فقط · التوليد يحترم خدمات العرض (عقد بلا زيارات = لا بنود زيارات) · اختبارات الترحيل الثلاثة · تسجيل كل أفعال الرابط في سجل النشاط.
